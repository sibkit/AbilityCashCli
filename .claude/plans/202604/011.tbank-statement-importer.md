# 011 — Импорт выписки Т-Банка + унификация банковских выписок

## Проблема

Выписка Т-Банка (`;`-CSV, UTF-8, русские заголовки) не распознаётся: `AlfaBankRule` требует TSV/CP1251 + английские маркеры, никакой другой rule не подходит → `BulkImportRunner` молча `skip` («нет importer'а»). Параллельно: skip без importer'а — это всегда сюрприз, должен быть фатальной ошибкой; счёт в выписке надо находить по суффиксу `[..xxxx]` в имени, а не через `BankStatementsConfig.AccountByRch`; общий слой банковских импортеров отсутствует — каждый банк дублирует writer.

## Риски

- **Парсер CSV вручную**: квотинг с `""` внутри легко сломать на edge-кейсах (перевод строки внутри поля). По выписке такого нет (поля однострочные), но если появится — упадём. Простое правило: «парсим построчно, разделитель `;`, поле в `"…"` при наличии `;` или `"` — `""` декодируем в `"`».
- **Конфликт суффиксов `[..xxxx]`**: если два счёта оканчиваются на одни 4 цифры (теоретически возможно) — `BankAccountResolver` должен падать с понятной ошибкой. Не пытаться угадывать.
- **Удаление `BankStatementsConfig`**: ломает существующий пользовательский `appsettings.json`. Альфа сейчас полагается на маппинг — после миграции счёт должен определяться по имени `[..xxxx]`. Проверить, что у боевых alfa-счетов суффикс уже стоит; если нет — пользователь правит имя счёта в AbilityCash до запуска.
- **Зарплатные строки в выписке** дадут визуальный «дубль» с импортом из реестра выплат — пользователь правит вручную; зафиксировано.
- **Регистронезависимость и пробелы** в заголовках Т-Банка: парсер должен `Trim` и сравнивать строго (без понижения регистра); если Т-Банк сменит формулировку шапки — импорт упадёт явной ошибкой.
- **Касса `[ФИО]`** и Enterprise→`[ЗП]` маппинг — отложены, не в этом плане.
