# 010 — Агрегация ошибок импорта

## Проблема

Правила и writers бросают `InvalidOperationException` на первой же проблеме, поэтому в `errored` попадает максимум одна ошибка на файл. Нужно собрать все ошибки (в т.ч. внутри одного файла), откатить транзакцию при любой и вывести человеко-читаемый список, сгруппированный по файлу; успешно импортированные файлы печатать зелёным.

## Файлы

- `AbilityCashCli/src/Import/ImportError.cs` — новый record: `File`, `Row?`, `Phase`, `Message`.
- `AbilityCashCli/src/Import/IImportRule.cs` — контракт `ExecuteAsync` возвращает `RuleResult(int RowsRead, int RowsSaved, IReadOnlyList<ImportError> Errors)`.
- `AbilityCashCli/src/Import/Rules/CashPayoutsRule.cs`, `VacationsRule.cs`, `TBankSalaryRegisterRule.cs`, `AlfaBankRule.cs`, `TimesheetRule.cs` — копить ошибки, не throw.
- `AbilityCashCli/src/Import/CashPayoutsWriter.cs`, `Vacations/VacationsWriter.cs`, `SalaryRegisters/SalaryRegisterWriter.cs`, `BankStatements/BankStatementWriter.cs`, `Timesheets/TimesheetWriter.cs` — `TryWriteRow`/аккумулятор ошибок вместо throw.
- `AbilityCashCli/src/Import/CategoryPathResolver.cs` — `TryResolve` рядом с `Resolve`.
- `AbilityCashCli/src/Import/BulkImportRunner.cs` — агрегировать ошибки правил, rollback если список непустой, не ловить через catch доменные ошибки.
- `AbilityCashCli/src/Cli/ImportReportWriter.cs` — новый: форматтер. Успех — зелёным (`ok: <file> (rows=..., saved=...)`), ошибки — блок на файл с разделителем и списком `[row N] phase: message`.
- `AbilityCashCli.Tests/src/Import/ImportErrorAggregationTests.cs` — несколько ошибок в одном файле собираются все; rollback при любой.

## Шаги

1. Ввести `ImportError` и `RuleResult`; обновить `IImportRule`.
2. В резолверах добавить `TryResolve*` варианты, возвращающие `null` + причину.
3. В каждом writer завести метод записи одной строки, возвращающий `ImportError?`; хранить список.
4. В каждом rule прогонять все строки, собирать `ImportError`, возвращать `RuleResult`.
5. Разделить фатальные ошибки (пустой файл, нечитаемый формат) — они по-прежнему исключения, но ловятся runner'ом и превращаются в один `ImportError(Phase="parse")`.
6. `BulkImportRunner`: собрать все `ImportError` от всех файлов; rollback если есть; печать через `ImportReportWriter`.
7. `ImportReportWriter`: зелёные ok-строки только при `errors.Count == 0`; при ошибках — заголовок файла, список, разделитель.
8. Тесты: unit на форматтер, интеграционный на rule с двумя плохими строками — обе в выводе.

## Риски

- Объём: 5 правил × 5 writers × резолверы — много точек, где `throw` заменяется на аккумулятор. Правки механические, но широкие.
- EF Core буфер: если writer добавил сущности в `DbContext` до ошибки и транзакция откатывается — ок, но если commit всё же пройдёт по другой ветке, частичные записи уйдут. Проверить, что ни одна ветка не коммитит при непустом списке ошибок.
- «Фатальные» ошибки парсинга трудно разделить с «строковыми» — начальная классификация может быть неточной; уточнять по ходу.
- Порядок ошибок: желателен стабильный (по строке источника), чтобы вывод был воспроизводимым.
