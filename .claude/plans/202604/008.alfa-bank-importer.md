# 008 — Импорт выписки Альфа-Банка

## Проблема
Выписки Альфа-Банка (CP1251, TSV с расширением `.csv`, формат 1С-Банк-Клиент) пропускаются импортом — нет importer'а. Нужно разложить все строки в целевой счёт AbilityCash как доходы (`d_c=C`) / расходы (`d_c=D`) без статьи; другие значения `d_c` и не-RUR — исключение.

## Файлы
- `src/Import/IImportRule.cs` — убрать свойства `Importer`/`Writer`, добавить `ExecuteAsync(source, path, ct)→(rowsRead, rowsSaved)`. Требуется, т.к. `ImportRecord` не подходит для банковских строк.
- `src/Import/BulkImportRunner.cs` — переключить на `rule.ExecuteAsync`; логирование `rows=`/`saved=` сохранить.
- `src/Import/Rules/CashPayoutsRule.cs`, `src/Import/Vacations/VacationsRule.cs`, `src/Import/SalaryRegisters/TBankSalaryRegisterRule.cs` — адаптация: внутри `ExecuteAsync` делегируют в старую пару Importer+Writer.
- `src/Import/BankStatements/AlfaBankRecord.cs` — DTO: Date, DC, AmountRur, Number, ODate, Text70, Rch, Currency, Counterparty(Name/Inn/Acc).
- `src/Import/BankStatements/AlfaBankImporter.cs` — CP1251 TSV парсер: детекция шапки по маркерным колонкам (`statement_unid`, `Rch`, `sum_rur`, `d_c`, `text70`), валидация `Type=RUR` и `d_c∈{D,C}`, `ru-RU` для чисел и `dd.MM.yyyy` для дат. Все `Rch` в файле обязаны совпадать.
- `src/Import/BankStatements/BankStatementWriter.cs` — маппинг `Rch → AccountName` из конфига → `Accounts.Name`. Для D: `ExpenseAccount`, для C: `IncomeAccount`; `TransactionCategory=null`. `Comment = "[{контрагент}] {text70}"` (контрагент — `pol_name` при D, `plat_name` при C, trim + сжатие пробелов). `ExtraComment1 = "№{number} от {o_date}, ИНН {counterparty_inn}"`. `ExtraComment2 = BuildSourceComment(...)`. Один `TransactionGroup` на файл.
- `src/Import/BankStatements/AlfaBankRule.cs` — `Matches`: `.csv` + первая строка в CP1251 содержит маркеры формата. `ExecuteAsync` — связка importer+writer.
- `src/Configuration/BankStatementsConfig.cs` — `IReadOnlyDictionary<string,string> AccountByRch`.
- `src/Configuration/AppConfig.cs` — секция `BankStatements`, дефолт в `CreateDefault`.
- `src/Program.cs` — регистрация в DI; `Encoding.RegisterProvider(CodePagesEncodingProvider.Instance)` при старте.
- `appsettings.json` (пример, если есть) — пара `"40702810323300000238": "Альфа (LM) [..0238]"`.

## Шаги
1. Рефакторинг `IImportRule`: убрать Importer/Writer свойства, ввести `ExecuteAsync`. Адаптировать 3 существующих rule и `BulkImportRunner`.
2. Зарегистрировать `CodePagesEncodingProvider` в `Program.cs`.
3. Добавить `BankStatementsConfig` + секцию в `AppConfig`/`CreateDefault`.
4. Реализовать `AlfaBankRecord`, `AlfaBankImporter` (парсер + валидация + сбор DTO).
5. Реализовать `BankStatementWriter` (резолв счёта, сборка `Transaction`, `TransactionGroup`).
6. Реализовать `AlfaBankRule` (`Matches` по первой строке CP1251, `ExecuteAsync`).
7. Зарегистрировать рулс/импортёр/писатель/конфиг в DI.
8. Обновить конфиг пользователя: добавить `BankStatements.AccountByRch`.

## Риски
- `AbilityCash` может не принять `TransactionCategory=null` для доход/расход. Проверить на одной записи до массового импорта; если упрётся — потребуется «пустая» категория-заглушка.
- Рефакторинг `IImportRule` трогает все существующие rule'ы. Нужна сквозная проверка импортов ФОТ/отпусков/реестров.
- Маркер формата (.csv + TSV в CP1251) слабый — `Matches` обязан читать строго первую строку и проверять полный набор маркерных колонок, иначе коллизия с будущими CSV.
- Имя владельца счёта в данных (`Name`) может отличаться от пользовательской метки (`Альфа (LM) [..0238]`) и от фактического банка (БСПБ vs Альфа) — резолвим **только по `Rch`**, остальные поля не используем для идентификации.
- Нормализация контрагентов — вне скоупа (отдельный план). Сейчас trim + collapse whitespace.
- Дедупликация между пересекающимися выписками — вне скоупа; контролирует пользователь.
