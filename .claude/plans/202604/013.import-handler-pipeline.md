# 013 — Двухэтапный pipeline импорта

## Проблема

`IImportRule.Matches` решает судьбу файла по одному признаку (имя или шапка) и first-match-wins. Для `.csv` с 20-значным счётом в имени `TBankStatementRule` перехватывает альфовский файл и падает в `ExecuteAsync`, хотя `AlfaBankRule` мог бы его распознать по шапке.

## Файлы

- `AbilityCashCli/src/Import/IImportRule.cs` → `IImportHandler.cs` — новый контракт: `bool MatchesByName(string path)` + `Task<HandlerResult?> TryImportAsync(string source, string path, CancellationToken ct)`.
- `AbilityCashCli/src/Import/ImportError.cs` — переименовать `RuleResult` в `HandlerResult` (в том же файле).
- `AbilityCashCli/src/Import/IImportRouter.cs` — сигнатура: `Task<HandlerResult?> ImportAsync(string source, string path, CancellationToken ct)`. `Resolve` убирается.
- `AbilityCashCli/src/Import/ImportRouter.cs` — реализация двухэтапного алгоритма (см. «Шаги»).
- `AbilityCashCli/src/Import/BulkImportRunner.cs` — вместо `router.Resolve + rule.ExecuteAsync` зовёт `router.ImportAsync`; ветка `rule is null` становится «результат = null».
- `AbilityCashCli/src/Import/Rules/CashPayoutsRule.cs` → `AbilityCashCli/src/Import/Handlers/CashPayoutsHandler.cs` (папка `Rules/` → `Handlers/`).
- `AbilityCashCli/src/Import/Vacations/VacationsRule.cs` → `VacationsHandler.cs`.
- `AbilityCashCli/src/Import/Timesheets/TimesheetRule.cs` → `TimesheetHandler.cs`.
- `AbilityCashCli/src/Import/SalaryRegisters/TBankSalaryRegisterRule.cs` → `TBankSalaryRegisterHandler.cs`.
- `AbilityCashCli/src/Import/BankStatements/Alfa/AlfaBankRule.cs` → `AlfaBankHandler.cs`.
- `AbilityCashCli/src/Import/BankStatements/TBank/TBankStatementRule.cs` → `TBankStatementHandler.cs`.
- `AbilityCashCli/src/Program.cs` — массив `IImportRule[]` → `IImportHandler[]`, новые имена классов.
- `CLAUDE.md` — секция «Роли файлов и привязки»: обновить с `IImportRule.Matches` на двухэтапный `IImportHandler`.

## Шаги

1. Переименовать `RuleResult` → `HandlerResult` в `ImportError.cs`.
2. Ввести `IImportHandler` с парой `MatchesByName` + `TryImportAsync`. Удалить `IImportRule`.
3. Переименовать и адаптировать 6 правил:
   - `MatchesByName`: только `TBankStatementHandler` возвращает `true` при 20-значном счёте в имени; остальные — `false`.
   - `TryImportAsync`: перенести проверку шапки/маркеров из старого `Matches` в начало метода. Если шапка не своя — `return null`. Иначе — текущая логика `ExecuteAsync`, возврат `HandlerResult`.
4. Переместить `Import/Rules/CashPayoutsRule.cs` → `Import/Handlers/CashPayoutsHandler.cs`, удалить папку `Rules/`.
5. Переписать `ImportRouter.ImportAsync`:
   - Этап 1: собрать `candidates = handlers.Where(h => h.MatchesByName(path))`. Пройти по ним по порядку; первый non-null результат `TryImportAsync` — вернуть.
   - Этап 2: если этап 1 пуст или все вернули null — пройти по **всем** `handlers` (исключая уже вызванных на этапе 1), первый non-null — вернуть.
   - Никто не взял → `return null`.
6. В `BulkImportRunner` заменить `router.Resolve + rule.ExecuteAsync` на `router.ImportAsync`. `null` → тот же «нет importer'а».
7. Обновить `Program.cs`: массив `IImportHandler[]` + новые имена.
8. Обновить CLAUDE.md.

## Риски

- DI собирается вручную в `Program.cs` — пропустить один handler = тихий «нет importer'а». Сверить список до/после.
- Порядок handlers в `Program.cs` влияет на этап 2 (first-match среди тех, кто распознал по шапке). `AlfaBankHandler` и `TBankStatementHandler` не конфликтуют по шапкам, но при добавлении новых банков порядок может стать значимым.
- В `ExecuteAsync` T-Bank было исключение «имя 20 цифр, но шапка не Т-Банк» — теперь этот случай легально передаётся дальше по цепочке (то есть ошибка превращается в «альфовский файл распознан»). Поведение меняется намеренно, не регресс.
